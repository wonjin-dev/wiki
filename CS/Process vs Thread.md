# 프로세스와 스레드

## 프로세스

자원이 메모리에 할당 받고 실행되는 단위<br>
웹 관점으로 보면 웹 애플리케이션 또는 웹 페이지를 지칭.<br>
각 프로세스는 독립된 메모리 공간을 가지고 있으며, 개별적으로 실행됨.<br>
때문에 각각의 프로세스는 메모리를 공유하려면 별도의 작업이 필요

- **_크롬 탭 하나당 프로세스 하나_**

## 스레드

프로세스의 자원을 할당받은 실행 단위<br>
웹 애플리케이션의 스레드는 프로세스의 일부로서 동시에 실행될 수 있다.<br>
스레드는 프로세스의 메모리를 공유하므로, 데이터 공유와 통신이 비교적 간단.<br>
프론트엔드에서 스레드는 웹 페이지의 로딩, 이벤트 처리, 애니메이션 등과 같은 작업을 동시에 처리하기 위해 사용.

## 멀티 프로세스와 멀티 스레드

`멀티 프로세스`와 `멀티 스레드`는 어플리케이션에 대한 처리방식 이다.
<br/>

### 멀티 프로세스

이름처럼 하나의 응용 프로그램에 여러 개의 프로세스를 실행하는 방식이다.<br/>
일반적으로 하나의 프로그램 실행에 대해 하나의 프로세스가 메모리에 생성된다. 그러나 멀티 프로세스를 사용한 경우 `하나의 부모 프로세스`가 여러 개의 자식 프로세스를 생성하여 다중 프로세스를 구성하는 구조이다.<br/>
각각의 프로세스는 고유햔 PID를 가지고 있다. 부모 프로세스는 자식 프로세스의 PID를 통해 제어할 수 있으며, 자식 프로세스는 부모의 PID(PPID)를 알고 있어, 통신이 가능하다.<br/>
(다만, 통신이 가능할 뿐, 엄연히 서로 다른 프로세스로 독립적인 메모리 공간을 가지고 서로 다른 작업을 독립적으로 수행한다)<br/>
<br/>
대표적인 예는 크롬 탭이다.
각 브라우저 탭은 같은 브라우저 프로그램 실행이지만, 각기 다른 어플리케이션을 실행한다.

#### 멀티 프로세스의 장점

##### 프로그램의 안정성

각 프로세스가 독립적인 메모리 공간을 가지므로, 한 프로세스가 비정상적으로 종료되어도 다른 프로세스에 영향을 주지 않는다.

##### 시스템 확장성

마찬가지로 각 프로세스가 독립적이므로 시스템의 규모를 쉽게 확장할 수 있다.<br/>
<br/>
node의 pm2 같은 서버 클러스터 구성을 생각하면 된다.

#### 멀티 프로세스의 단점

##### 성능 저하의 위험

티 프로세스를 구성하여 성능이 저하되는 것은 당연히 아니다.<br/>
<br/>
멀티 태스킹을 위한 프로세스를 컨텍스트 스위칭 하면,<br/> 스위칭 될 때마다

1. 변경될 프로세스의 메모리를 탐색하고,
2. 캐시 메모리 초기화,
3. 프로세스 상태 저장 등 불러올 데이터를 준비해야 한다.

때문에 빈번한 컨텍스트 스위칭으로 오버헤드가 발생할 수 있게 된다.<br/>
(반면 스레드를 컨텍스트 스위칭하면 프로세스 스위칭 보다 가벼워 훨씬 빠르고 좋다)
<br/>
<br/>
따라서, 멀티 프로세스 환경에서는 컨텍스트 스위칭을 최소화하는 방법이 중요하다.<br/><br/>
이를 위해

- 프로세스 수를 적정하게 유지
- I/O 바운드 또는 CPU 바운드 작업이 많은 프로세스를 분리하여 관리
- CPU 캐시를 효율적으로 활용

등 최적화 방법을 고려해야한다

#### 비효율적인 자원 공유

만일 각 프로세스간에 자원 공유가 필요할 경우 IPC를 활용해야 한다.
<br/>
하지만 IPC 자체로 오버헤드가 발생한다.<br/>
(데이터를 복사하거나 버퍼링해야 하기 때문에)

## 멀티 스레드

멀티 스레드는 하나의 프로세스 안에 여러 개의 스레드가 있는 것을 말한다.<br/>
따라서 하나의 프로그램에서 두가지 이상의 동작을 동시에 처리할 수 있게된다.
<br/>
(JS 싱글 쓰레드와 promise, 콜스택의 PTSD..)

### 멀티 스레드의 장점

#### 프로세스보다 가벼움

당연하지만 스레드는 프로세스 내에서 생성되기 때문에 스레드의 실행 환경을 설정하는 작업이 매우 간단하다.<br/>
(생성 및 종료가 빠르다)<br/>

#### 효율적인 자원 공유

스레드는 코드, 데이터, 스택 영역을 제외한 나머지 자원을 서로 공유하기 때문에 기본적으로 내장된 데이터 용량이 프로세스보다 작다.<br/>
보다 효율적으로 자원을 공유한다.

#### 컨텍스트 스위칭 비용 감소

스레드에도 컨텍스트 스위칭 오버헤드가 존재한다. <br/>
하지만 상대적으로 프로세스 컨텍스트 스위칭보다 비용이 훨씬 낮다.

### 멀티 스레드의 단점

#### 상대적으로 낮은 안정성

프로세스가 독립적인 메모리와 실행 환경으로 애플리케이션의 안정성을 확보했다.<br/>많은 자원을 공유하는 스레드는 당연하게도 하나의 스레드에 문제가 생겼을 때 어플레케이션 자체에 영향을 줄 확률이 높아진다.
<br/><br/>
물론 적절한 예외 처리나 스레드 풀에서 잔여 스레드를 가져오던지 하여 안정성을 높일 수는 있다.
<br/>
하지만 이 경우에는 스레드 처리에 추가 비용이 발생한다.

**_스레드 풀_**
스레드를 미리 생성하고, 작업 요청이 발생할 때 마다 미리 생성된 쓰레드로 해당 작업을 처리하는 방식<br/>
(작업이 끝난 쓰레드는 종료되지 않으며 다음 작업 요청이 들어올때까지 대기)

#### 동기화로 인한 성능 저하

멀티 스레드는 여러 개의 스레드가 공유된 자원에 동시에 접근할 수 있기 때문에, 동기화 문제가 발생할 수 있다.<br/>
때문에 여러 스레드들이 자원에 대한 접근을 순차적으로 통제해야한다.<br/>
그러나 동기화 작업은 여러 스레드 접근을 제한하는 것이기 때문에 병목 현상이 일어나 성능이 저하될 가능성이 높다.<br/>
(이를 해결하기 위해 임계 영역에 대하여 뮤텍스 또는 세마포어 방식을 활용한다)
<br/><br/>

**_임계 영역 (Critical Section)_**
멀티 스레드에서 임계 영역은 공유되는 자원을 접근하는 코드 영역을 말한다.<br/>
대표적으로 전역 변수, heap 메모리 영역이다.
<br/>

**_뮤텍스 (Mutex)_**
공유 자원에 대해 오직 1개의 스레드만이 공유 자원에 접근할 수 있도록 제어하는 기법이다<br/>
임계 영역에 진입하기 전에 락을 획득하고, 임계 영역을 빠져나올 때 락을 해제하여 다른 스레드들이 접근할 수 있도록 한다.
<br/>

**_세마포어 (Semaphore)_**

동시에 접근 가능한 스레드의 개수를 지정하는 기법이다.<br/>
스레드가 임계 영역에 진입하기 전에 세마포어 값을 확인하고, 값이 허용된 범위 내에 있을 때만 락을 획득할 수 있는 형식이다.

<br/><br/>

#### 어려운 디버깅

여러 개의 스레드가 동시에 실행되기 때문에, 각 스레드의 동작을 추적하기 어려울 수 있다.

<br/><br/>

### 데드락 (교착상태)

공유자원 동시 엑세스로 인해 발생하는 문제이다.<br/>
다수의 프로세스나 스레드가 서로 자원을 점유한 상태에서 각자가 점유한 자원을 기다리는 상황. 즉, 교착 상태를 말한다.<br/>
(여러 개의 스레드가 서로 대기하면서 무한정 기다리게되는 무한 루프와 같은 증상)
