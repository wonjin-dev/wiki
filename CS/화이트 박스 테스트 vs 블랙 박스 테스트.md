# 화이트 박스 테스트

> 소프트웨어의 내부 구조, 설계, 구현을 알고 있는 상태에서 테스트하는 방법

소스 코드, 알고리즘, 데이터 흐름 등을 직접 검토하여 `코드의 논리적 정확성`을 테스트한다.

## 특징

- **코드 기반 테스트**
  > 내부 코드 로직을 기반으로 테스트 케이스를 설계
- **경로 테스트**
  > 가능한 모든 코드 경로를 테스트해서 모든 조건과 분기를 검증.
- **루프 테스트**
  > 루프 구조의 초기화, 조건, 종료 등을 검토해서 무한 루프 등을 방지
- **조건 테스트**
  > 조건문(if, switch 등)이 올바르게 동작하는지 확인

## 장점

- 내부 코드 구조를 완벽히 이해하고 테스트할 수 있다.
- 코드 커버리지가 높아서 잠재적인 버그를 쉽게 발견할 수 있다.
- 모든 경로를 테스트한다.
  - 논리적 오류를 찾기 쉽다.

## 단점

- 코드에 대한 깊은 이해가 필요하다.
- 대규모 소프트웨어의 경우 모든 경로를 테스트하는 게 어렵고 비용이 많이 든다.
- 내부 구조가 자주 변경될 경우 테스트 케이스를 계속 업데이트해야 한다.

# 블랙 박스 테스트

> 소프트웨어의 내부 구조나 동작 원리를 알지 못한 상태에서 외부 기능만 테스트하는 방법

사용자가 기대하는 기능을 올바르게 수행하는지 확인하는 데 중점을 둔다.

## 특징

- 주로 사용자 관점에서 소프트웨어의 입력과 출력을 검증한다.
- **기능 기반 테스트**
  > 소프트웨어의 기능 요구사항을 기반으로 테스트 케이스를 설계
- **경계값 분석**
  > 입력 값의 경계에서 오류가 발생할 가능성이 높으니까 경계값을 중점적으로 테스트
- **등가 분할 테스트**
  > 입력 데이터의 범위를 여러 등가 클래스(동일한 결과를 도출하는 데이터 그룹)로 나눠서 테스트
- **결정 테이블 테스트**
  > 여러 조건과 그에 따른 행동을 테이블 형식으로 정리해서 테스트
- **상태 전이 테스트**
  > 소프트웨어의 상태 변화와 그에 따른 동작을 검증

## 장점

- 사용자 관점에서 소프트웨어의 기능을 검증할 수 있다.
- 내부 코드 구조를 몰라도 테스트할 수 있어서 비기술자도 테스트에 참여할 수 있다.
- 비교적 간단하게 테스트 케이스를 설계할 수 있다.

## 단점

- 내부 코드 구조를 검증하지 않아서 모든 버그를 발견하기 어렵다.
- 주로 외부 동작만 테스트하니까 코드 커버리지가 낮다.
- 입력 값 조합이 많을 경우 모든 경우를 테스트하는 게 현실적으로 불가능하다.

# 요약

| 특징        | 화이트박스 테스트          | 블랙박스 테스트              |
| ----------- | -------------------------- | ---------------------------- |
| 테스트 기반 | 코드 로직                  | 기능 요구사항                |
| 테스트 관점 | 개발자                     | 사용자                       |
| 장점        | 높은 코드 커버리지         | 사용자 관점의 기능 검증 가능 |
| 단점        | 코드에 대한 깊은 이해 필요 | 내부 구조 검증 불가          |
| 적용 범위   | 내부 논리 및 경로          | 외부 동작 및 인터페이스 검증 |
